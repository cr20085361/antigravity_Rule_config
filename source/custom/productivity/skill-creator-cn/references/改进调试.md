# 改进调试：E5 诊断剧本

本文件处理 **E5**——用户已有一个 skill，但它"不好用"或"不触发"。与从零创建不同，这里的核心逻辑是：**先读懂现有 skill，定位症状，再有针对性地修改**。盲目改动往往越改越乱，诊断清楚才能一次命中。

---

## 一、入口：先读懂现有 skill

在动手之前，把以下材料全部通读一遍：

- `SKILL.md`（frontmatter 里的 `name` 和 `description`，以及正文的指令逻辑）
- `references/` 下的所有文件（了解 skill 依赖的知识结构）
- 如果有 `evals/`，扫一眼已有的测试用例

读完后，问用户一个问题把症状锁定到一个类别：**"你观察到的问题是什么？是根本没被触发、触发了但输出不对、触发太频繁/太少，还是输出太啰嗦？"**

不要等用户自己分类——给他选项，他选完你就有了诊断起点。

---

## 二、诊断决策树

症状 → 检查点 → 动作，三步走。

---

### 症状 A：不触发

**"我说了好几次，skill 就是没反应。"**

**检查点一：description 字段**

打开 SKILL.md 的 frontmatter，读 `description` 字段，问自己：
- 有没有明确说"做什么"（skill 的核心能力）？
- 有没有明确说"何时用"（触发场景或触发短语）？
- 描述是否太窄——只命中某一个非常具体的措辞，而用户平时不会这么说？
- 描述是否太泛——说了等于没说，和其他 skill 或 Claude 本身的能力完全重叠？

触发机制的本质：Claude 根据 `name + description` 判断要不要调用这个 skill。描述写得不清或不够"推销自己"，skill 就会被跳过。另外，太简单的单步任务（比如"读这个文件"）本来就不会触发 skill——Claude 会直接用基础工具处理，这是正常行为，不是 bug。

**动作：走描述优化流程**

在 `references/重型评测.md` 的「Description 触发优化」一节里，有一套完整的 `scripts/run_loop.py` 优化流程：生成触发测评集 → 用户确认 → 后台运行优化循环 → 选出最优 description → 更新 frontmatter。这是专门解决"不触发"问题的标准路径，效果比手动猜测好得多。直接走那个流程。

---

### 症状 B：行为跑偏 / 输出不对

**"触发了，但做出来的不是我想要的。"**

**检查点一：指令是否含糊**

读 SKILL.md 正文的核心指令段，问自己：
- 有没有说清楚输出的格式、长度、语气？
- 有没有解释"为什么这么做"——只有 MUST/NEVER，没有理由？光靠禁令，模型在边界情况下容易失去判断力；解释背后的原因，模型才能在新场景里举一反三。
- 有没有提供具体例子或参考样本？对于有固定输出格式的 skill，一两个例子比两百字的描述更有效。

**检查点二：references 是否缺位**

如果 skill 依赖某个领域知识（格式规范、专业术语、上下文背景），但这些没有放进 `references/`，模型只能靠训练知识猜测，稳定性很差。检查：有没有应该放进 references 但没放的材料？

**动作：定点改写**

找到含糊指令，把它重写成"目标 + 理由 + 示例"三段式。不要堆砌更多禁令，而是解释清楚为什么某种做法是对的——这样模型在遇到没见过的输入时也能做出正确判断。如果需要添加 references，在 SKILL.md 里明确标注何时应该去读那个文件。

---

### 症状 C：过触发 / 欠触发

**"明明不该用这个 skill，它却触发了"，或者"有时触发，有时不触发，很随机。"**

**检查点：description 的边界是否清晰**

过触发通常意味着 description 覆盖了太宽的范围，和相邻 skill 或 Claude 的通用能力产生了模糊地带。
欠触发（随机）通常意味着描述里缺少足够多样的触发短语，或者触发条件太依赖某个特定关键词。

**动作：描述优化 + 理解触发机制**

同样走 `scripts/run_loop.py` 的优化流程，但在生成测评集时要重点设计"近似场景"的负例——那些用了相似词汇但不该触发的情况。理解底层机制也很重要：Claude 只在认为任务足够复杂、需要专门指导时才触发 skill；如果某个场景本来就是简单的单步操作，期望 skill 稳定触发是不现实的，这时候考虑把触发场景改成更复杂的版本，或者重新评估这个场景是否真的需要 skill。

---

### 症状 D：太重 / 太啰嗦

**"每次 skill 跑完，过程太冗长，输出里有大量我不需要的内容。"**

**检查点：指令是否过度堆砌**

- 有没有大量类似"在做 X 之前，先确认 Y，然后再检查 Z，最后才能进行 W"这种过度细化的步骤？
- 有没有大段的背景解释被塞进 SKILL.md 正文，而它们只在少数情况下才用到？
- SKILL.md 有没有超过 500 行？如果有，几乎肯定存在可以下沉到 references 的内容。

**动作：精简 + 分层**

删掉不拉车的指令——即那些不改也不会影响核心输出的指令。把只在特定场景下才需要的细节移到 `references/` 里，在 SKILL.md 里留一行"如遇 X 情况，读 `references/xxx.md`"。SKILL.md 应该像一份清晰的路线图，而不是一本操作手册。精简之后，模型注意力更集中，输出反而更精准。

---

## 三、改完即验证

每轮改动之后，不要靠直觉判断"应该好了"——用分层验证来确认。

**默认用轻量验证：** 构造 2–3 个代表性的测试 prompt，直接用改后的 skill 跑一遍，给用户看输出。这是 SKILL.md 主流程里"运行测试用例"的轻量版，快速、直观、成本低。

**重量级评估按需启动：** 如果用户需要更严格的对比（比如"我要知道改完之后到底比之前好多少"），走完整的评估循环——建 `evals.json`、并行跑 with-skill 和对照版本、生成 benchmark、打开 eval viewer 让用户审阅。这个流程在 `references/重型评测.md` 的「重型量化评测」一节里有完整说明。

**description 改动后的专项验证：** 如果这次修改包含了 description 字段的更新，在验证完输出质量后，额外跑一轮 `scripts/run_loop.py` 确认触发准确率有提升。

---

## 四、保留原名与就地修改注意事项

改现有 skill 时，有两件事容易踩坑：

**保留目录名和 `name` 字段不变。** skill 的身份由目录名和 frontmatter 里的 `name` 字段共同决定。如果改名，安装路径会变、旧版引用会断、用户的肌肉记忆也会失效。无论做多大的改动，`name` 字段和目录名都应该保持原样。比如 skill 原来叫 `research-helper`，改完打包后仍然输出 `research-helper.skill`，不要变成 `research-helper-v2`。

**只读路径先拷到可写位置再编辑。** 已安装的 skill 路径可能是只读的，直接在原路径编辑会报权限错误。正确做法是：把 skill 目录完整拷贝到 `/tmp/skill-name/`，在那里做所有修改，打包后再覆盖回去。这个细节在 SKILL.md 的"平台差异说明 → Claude.ai"一节（更新已有 skill 的要点）有说明，不要跳过。

---

**诊断完成后的交接：** 四个症状类别都定位清楚了，就可以回到 SKILL.md 的对应流程——description 问题走优化脚本，指令问题走改写 + 轻量验证，改完打包交给用户。
