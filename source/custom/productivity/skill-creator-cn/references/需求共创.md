# 需求共创：挖掘引擎

本文件承载 E1（模糊想法）、E2（从对话提取）、E3（现有材料）、E4（已有草稿）四类入口的需求挖掘方法。目标是：从任意起点出发，通过对话把用户的真实需求收敛到"四要素"，再交由 SKILL.md 的写作流程接手。

---

## 一、四要素框架

任何入口都要收敛到这四个要素。它们不是凑数的——分别对应 skill 运行机制的四个关键维度：

| 要素 | 问的问题 | 对应 skill 机制 |
|---|---|---|
| ① 核心能力/产出 | 让 Claude 做成什么？最终给用户的是什么？ | skill 的行为主体 |
| ② 触发时机 | 用户会怎么说？在什么场景下才该启动？ | description 字段（触发器） |
| ③ 输入与输出 | 拿什么进来、交出什么？格式、结构、长度？ | IO 协议与格式约定 |
| ④ 边界与护栏 | 不该做什么？哪些是常见误用或易出错点？ | 保护机制与范围限定 |

为什么是这四个？因为 skill 的失败模式几乎都能归结到其中某个维度的缺失——要么不知道该交出什么，要么触发时机模糊导致乱触发或漏触发，要么 IO 格式不对接，要么没有护栏导致越界。把这四个锁死，后面写 SKILL.md 时就有清晰的骨架可以填。

---

## 二、对话原则

**一次一问。** 不要一次抛出三个问题。用户的认知带宽有限，同时面对多个问题时，往往只回答其中最容易的一个，或者给出含混的打包答复。一次一问，才能得到真正有用的回答。

**给选项，少让用户凭空想。** "输出格式你想要什么？"比不上"输出是一段话、一个列表、还是一份文件？"后者降低了用户的创作负担，也更容易收到精准的回答。

**先发散，再收敛。** 前几轮让用户自由描述，不急着纠正或聚焦。等你对用户的核心诉求有了感觉，再开始帮他收窄和精炼。

**每步确认，再推进。** 每个要素确认完毕再问下一个。这样用户知道自己在哪一步，也给了他纠正的机会，避免在一个错误理解上越走越远。

**跟用户的语言走。** 用户说中文就中文，切换到英文就英文。不要坚持用自己习惯的语言。

**发散时给 2–3 个具体方案，别只抛开放问题。** 当 skill 的范围或形态还不明确时，与其问"你想要什么样的"，不如给 2–3 种做法、各自的取舍，再点出你的推荐。比如："这个可以做窄一点（只整理成表格），也可以宽一点（顺带按时提醒）——窄的上手快、触发稳；宽的省事但容易触发不准。我建议先做窄的，你觉得呢？" 用户面对具体选项，比面对空白更容易说出真实偏好，这也是挖出隐藏需求最快的路径。

**YAGNI：先收敛成最小可用的 v1。** 主动砍掉"顺便还能……"的附加诉求，先把一个核心能力做扎实，其余记成后续迭代方向。一个 skill 聚焦一件事，触发才准、维护才轻；想一次做全，往往哪件都做不好。这与前置 Triage 的"聚焦单一核心能力"一脉相承。

## 三、水平自适应

全程不打断问"你是几级用户"。靠对话中的信号动态判断，随时调整措辞和问题颗粒度。

| 观察到的信号 | 判断 | 对话调整 |
|---|---|---|
| 用具体文件名、工具名、数据结构、字段名（比如 `.xlsx`、`pandas DataFrame`、`user_id` 列） | 偏专业 | 可以用术语，问题可以精炼直接，必要时可跳过"给个例子"环节直接进结构化澄清 |
| 自己主动给出了输入输出样例 | 偏专业 | 直接围绕这个样例确认转换规则，不必再从头问"你期望输出是什么" |
| 用"那种东西""差不多""不知道咋说""类似 XX 那感觉" | 偏小白 | 换成大白话 + 类比；把问题拆成更小的步骤；给具体选项让他挑，而不是等他自己描述 |
| 把 skill 当万能工具，想塞进大量不同场景 | 需引导收敛 | 先帮他聚焦到一个最核心的能力，其他的可以作为后续迭代方向；解释"一个 skill 做好一件事比做十件事更可靠"的原因 |
| 直接粘贴代码、配置文件、或者用 API 术语描述 | 开发者 | 可以讨论实现细节，可以直接问"触发器你想用什么短语"之类的技术问题 |

水平判断不是一次性的——随着对话推进，用户展示出新的信号时随时更新判断。

## 四、各入口处理

### E1：模糊想法

用户只有一个模糊的念头，比如"我想做一个帮我写周报的 skill"。

1. **从核心产出问起。** "这个 skill 最后交给你的是什么？一段文字、一个文件、还是直接帮你执行某个操作？" 不要从触发条件问起——用户往往对产出更有直觉。

2. **用选项帮用户锁定产出形态。** 如果用户说"就是那种……写好的东西"，给他选项："是一段可以直接复制粘贴的周报正文，还是一个填好内容的文档文件，或者是先问你几个问题再生成？"

3. **问"你现在怎么做这件事"。** 让用户描述现有流程，往往能挖出他自己没意识到的输入源（比如他会先看 Jira、再翻聊天记录）。

4. **逐步补齐四要素。** 核心产出确认后，按顺序问触发时机 → 输入格式 → 边界（什么情况下不该跑这个 skill）。每个要素一问，确认再推进。

5. **给具体例子让用户挑，而非让他凭空构想。** "触发词可以是'帮我写周报'、'整理本周工作'、或者你自己说的某个固定短语，哪个最自然？"

6. **四要素补齐后，走第六节的「动手前的确认门」，获批再起草。**

### E2：从对话提取

用户说"把我们刚才做的事情变成一个 skill"，或者对话里已经有一个完整的工作流。

1. **先从当前对话记录里挖，不重新盘问。** 梳理：对话里用到了哪些工具或步骤？有没有明显的顺序？用户在哪些地方纠正过你或强调过某个做法？

2. **把挖出来的内容映射到四要素，然后告诉用户你的理解。** 例如："我理解这个 skill 是：你给一个 CSV 文件路径 → skill 分析数据分布 → 给你一份摘要报告。触发词可能是你说'分析一下这份数据'或类似表达。这样理解对吗？"

3. **只就缺口提问。** 如果输入输出已经从对话里能看清楚，就不再问这些；只问真正缺失的部分，比如"边界条件我没从对话里看出来——有没有你不想让这个 skill 处理的情况？"

4. **确认完整四要素后，走第六节的「动手前的确认门」，获批再交由写作环节。**

### E3：现有材料·流程或文档

用户给了一份 SOP、操作手册、流程图（以文字形式粘贴），或者一段描述某个工作流程的文字。

1. **阅读材料，抽取以下维度：**
   - 步骤序列（做什么、按什么顺序）
   - 每步用到的工具或执行的动作
   - 判断分支（如果 A 则做 X，否则做 Y）
   - 用户在材料中特别强调或反复提到的点（往往是之前踩过坑的地方）

2. **把抽取结果回填到四要素框架，告知用户你的理解。** 列出你推断的核心能力、可能的触发场景、输入输出形态。

3. **请用户补缺。** 材料里通常缺边界和护栏，主动问："这份流程文档里有没有哪些特殊情况是你不想让 skill 自动处理、需要人工介入的？"

4. **如果材料里有含混或矛盾的地方，逐条澄清，不要自行假设。**

### E3：现有材料·输入→输出例子

用户给了一对或几对输入→输出样本，希望 skill 能复现这种转换。

1. **分析样本，反推转换规则：**
   - 输出格式的模板是什么（固定结构、固定字段、固定语气）？
   - 哪些部分是从输入直接映射过来的，哪些是加工或生成的？
   - 有没有隐含约束（比如长度限制、语言风格、数字精度）？

2. **主动追问：这是典型案例还是边界情况？** 这一步至关重要——单样本容易让 skill 过拟合到一个特殊情况上，导致在普通输入上表现差。问用户："这个例子算是最常见的输入，还是一个比较特殊的情况？能再给一个更普通的例子吗？"

3. **根据规律推断边界。** 问："如果输入里缺少某个字段，skill 应该怎么做——报错、跳过、还是填默认值？"

4. **把反推出的规则明确给用户确认，再走第六节的「动手前的确认门」进入写作。**

### E4：已有草稿

用户已经写了一版 SKILL.md 或者某种指令草稿，想让你帮他完善。

1. **先读懂草稿，理解它想做什么，不要急着改。**

2. **对照四要素找缺口：**
   - 核心产出是否清晰描述了？
   - 触发条件（description 字段）是否具体、有区分度？
   - 输入输出格式有没有明确约定？
   - 有没有护栏或边界说明？

3. **直接进打磨循环，不从头盘问。** 只就缺口向用户确认，而不是重新过一遍所有问题："草稿里我看到触发条件和产出都比较清楚，但边界部分是空白——有什么情况你不希望这个 skill 自动处理？"

4. **给出具体的改进建议（而不是泛泛评价），逐条确认后走第六节的「动手前的确认门」，再执行。**

## 五、常见卡点破解话术

**卡点一：用户说不清产出是什么**

用户给的描述是"就是帮我处理一下那个事情"或者"让它更好"。

处理方式：从用户的终态倒推。"如果这个 skill 做得很好，你下一步会用这个结果做什么？"——用户对用途的描述往往比对产出的描述更清晰，从用途反推产出形态通常更有效。如果还是模糊，给三个具体选项让他选："你拿到的会是：一段文字可以直接用？还是一个文件可以下载？还是 Claude 直接帮你执行某个操作？"

**卡点二：需求太大，用户想把所有场景都塞进一个 skill**

用户描述了七八个不同的使用场景，或者说"什么情况都可以用"。

处理方式：帮用户做优先级排序，而不是直接说"太多了做不了"。"这几个场景里，哪一个是你最频繁遇到、最希望先解决的？我们先把这个做好，其他的可以在第一版跑通后再扩展。一个 skill 聚焦一件事，触发更精准，效果也更稳定。" 解释原因比直接拒绝更容易被接受。

**卡点三：用户给的例子太少，或者只有一个**

用户只给了一对输入输出，希望 skill 以此为标准。

处理方式：主动追问，防止过拟合。"这个例子很有帮助，我想再多了解一下——这是你最常见的输入类型，还是一个比较特殊的情况？如果是常见情况，能再给一个不同的例子，帮我确认规律是否一致？如果是特殊情况，能给我看一个更普通的输入长什么样？" 目的是让 skill 学到规律，而不是记住单个案例。

## 六、动手前的确认门

四要素都填齐、方案也收敛之后，**不要直接开写 SKILL.md**——先过一道确认门。

把你对需求的理解回述成一页"需求小结"，覆盖四要素（核心产出 / 触发时机 / 输入输出 / 边界），然后明确问用户：**"我的理解是这样——准确吗？要补充或修正吗？没问题我就开始写了。"**

为什么这道门重要：需求是在几轮对话里逼近的，难免有偏差。动手前花一句话确认，比写完一版才发现跑偏、推倒重来便宜得多。这是把"每步确认"升级成交给起草环节之前的一道正式关卡——**理解未对齐，绝不开写**。

用户明确点头后，才进入起草（见 SKILL.md「③ 共有打磨循环 → 起草 SKILL.md」）。若用户在这一步又冒出新需求，回到对应入口补挖，再回到这道门。
