# 重型评测参考手册

本文件收录进阶可选的重型评测流程，供技术型用户按需查阅。

**何时来这里：仅当用户是技术型老手、主动要求严格量化对比、或要优化触发率时。**

---

## 重型量化评测

> 仅在以下情形启用：用户是技术型老手，或主动要求严格评测，或需要跨版本做量化对比。默认绝不把 JSON assertion、benchmark、eval-viewer 等概念主动抛给初学用户。

把结果存入 `<skill-name>-workspace/`（skill 目录的同级），按迭代组织：`iteration-1/`、`iteration-2/`……，每个迭代下按测试用例建子目录（`eval-0/`、`eval-1/`……）。

**第 1 步：同一轮并发启动所有 run（带 skill + baseline）**

每个测试用例在同一轮里启动两个子代理——一个带 skill，一个不带。不要先跑带 skill 的，再回来补 baseline——一起发出去，同时收结果。

带 skill 的 run：
```
Execute this task:
- Skill path: <path-to-skill>
- Task: <eval prompt>
- Input files: <eval files if any, or "none">
- Save outputs to: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- Outputs to save: <用户关心的输出，如 ".docx 文件"、"最终 CSV">
```

Baseline run（依场景而定）：
- 创建新 skill：不带任何 skill，存到 `without_skill/outputs/`。
- 改进已有 skill：用旧版本做 baseline（先快照：`cp -r <skill-path> <workspace>/skill-snapshot/`），存到 `old_skill/outputs/`。

为每个测试用例写 `eval_metadata.json`（assertion 暂时为空）：
```json
{
  "eval_id": 0,
  "eval_name": "描述性名称",
  "prompt": "用户的任务描述",
  "assertions": []
}
```

**第 2 步：run 进行中，起草 assertion**

别干等——这时候可以拟写每个测试用例的量化 assertion，向用户解释检查什么。好的 assertion 客观可验证，名称描述性强，一眼能看出在检查什么。主观类 skill（写作风格、设计质量）用定性评估，别强行写 assertion。

拟好后更新 `eval_metadata.json` 和 `evals/evals.json`。

**第 3 步：run 完成时，保存计时数据**

每个子代理任务完成时，通知里会带 `total_tokens` 和 `duration_ms`，立即存入 run 目录的 `timing.json`：
```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```
这数据只在通知里出现一次，不会持久化——逐条处理，别攒起来批量处理。

**第 4 步：评分、汇总、启动 viewer**

全部 run 完成后：

1. **评分**——启动 grader 子代理（读 `agents/grader.md`），评估每条 assertion 是否通过，结果存到 run 目录的 `grading.json`。grading.json 的 expectations 数组字段必须是 `text`、`passed`、`evidence`（不是 `name`/`met`/`details`），viewer 依赖这些字段名。能用脚本自动检查的，写脚本运行，比肉眼看更可靠、可复用。启动 grader/analyzer/comparator 子代理时，告诉它们用用户当前的语言输出 evidence/reasoning/笔记 等自由文本（默认中文）。

2. **汇总 benchmark**——在 skill-creator-cn 目录下运行：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   生成 `benchmark.json` 和 `benchmark.md`，包含每种配置的通过率、耗时和 token 数，以及均值 ± 标准差和差值。手动生成 benchmark.json 时，schema 见 `references/schemas.md`。with_skill 版本排在对应 baseline 前面。

3. **分析师视角**——读 benchmark 数据，发现汇总统计掩盖的规律。参考 `agents/analyzer.md`（"Analyzing Benchmark Results" 一节），关注：无论 skill 在不在都必然通过的 assertion（无区分度）、高方差 eval（可能不稳定）、时间/token 权衡。

4. **启动 viewer**：
   ```bash
   nohup python <skill-creator-cn-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   第 2 轮起，加 `--previous-workspace <workspace>/iteration-<N-1>`。

   无显示器环境（Cowork/headless）：用 `--static <output_path>` 输出独立 HTML 文件，用户点"Submit All Reviews"后会下载 `feedback.json`，把它复制进 workspace 目录供下一轮使用。

   请使用 `generate_review.py` 生成 viewer，不要自己写定制 HTML。在用户审阅结果、提供反馈之前，不要自己评估输出、擅自修改 skill。

5. **告知用户**，类似："我已在浏览器里打开了结果，有两个标签页——'Outputs' 可以逐个查看测试用例并留下反馈，'Benchmark' 显示量化对比。看完告诉我。"

Viewer 里用户看到的内容：
- **Outputs 标签页**：每次展示一个测试用例（prompt / 输出 / 上次输出对比 / 形式评分 / 反馈输入框）
- **Benchmark 标签页**：通过率、耗时、token 用量的统计汇总，以及分析师观察

看完后点"Submit All Reviews"，所有反馈存入 `feedback.json`。

**第 5 步：读取反馈**

用户完成后，读取 `feedback.json`：
```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少坐标轴标签", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完美，很喜欢", "timestamp": "..."}
  ],
  "status": "complete"
}
```
反馈为空表示用户觉得没问题。重点改用户有具体意见的用例。

完成后关掉 viewer 服务：
```bash
kill $VIEWER_PID 2>/dev/null
```

## 盲评对比

> 在需要更严格地对比两个版本的 skill 时使用（例如用户问"新版本真的更好吗"）。

基本原理：把两份输出交给一个独立子代理，不告诉它哪个是哪个，让它判断质量，然后分析胜出者为何胜出。读 `agents/comparator.md` 和 `agents/analyzer.md` 了解具体操作。

这是可选步骤，需要子代理，大多数用户不需要。人工审阅循环通常已经足够。

## Description 触发优化（详细流程）

description 字段是 Claude 决定是否调用 skill 的核心机制。skill 做好后，可以主动提出优化 description 以提升触发准确率。

### 第 1 步：生成触发评测查询集

生成 20 条评测查询——应触发和不应触发各占一半。保存为 JSON：

```json
[
  {"query": "用户的提示词", "should_trigger": true},
  {"query": "另一条提示词", "should_trigger": false}
]
```

查询要真实，像 Claude Code 或 Claude.ai 的真实用户会写的那种——具体、有细节，有文件路径、职位背景、列名、公司名、URL，有一点前情提要。可以有小写、缩写、错别字、口语。长短要多样，聚焦边缘情况，不要出太明显的题（用户有机会确认）。

**应触发（8–10 条）**：覆盖不同表达方式——正式的、口语的；用户没有明确提到 skill 名称或文件类型但明显需要的情况；本 skill 与其他 skill 竞争但应该赢的情况。

**不应触发（8–10 条）**：最有价值的是"险些命中"——和本 skill 共用关键词但实际需要别的东西，语义模糊的表述，以及触碰了 skill 功能但场景不对的情况。避免出太简单的负例（比如针对 PDF skill 写"写个斐波那契函数"，这种测不出什么）。

### 第 2 步：和用户确认查询集

用 HTML 模板展示查询集给用户审阅：

1. 读模板 `assets/eval_review.html`
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → JSON 数组（不加引号，是 JS 变量赋值）
   - `__SKILL_NAME_PLACEHOLDER__` → skill 名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 当前 description
3. 写到临时文件并打开：`open /tmp/eval_review_<skill-name>.html`
4. 用户可编辑查询、切换应触发/不应触发、增删条目，完成后点"Export Eval Set"
5. 文件下载到 `~/Downloads/eval_set.json`——如果有多个同名文件，取最新的（如 `eval_set (1).json`）

这一步很重要——查询集质量差，优化出来的 description 也会差。

### 第 3 步：运行优化循环

告知用户："这需要一些时间，我在后台运行优化循环，会定期给你进度更新。"

把查询集保存到 workspace，然后在后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

`--model` 使用当前会话的模型 ID（来自系统提示），确保触发测试和用户真实体验一致。

运行期间定期 tail 输出，给用户更新当前迭代和评分情况。

脚本自动处理整个优化循环：把查询集按 60% 训练集 / 40% 留出测试集拆分，评估当前 description（每条查询跑 3 次取可靠触发率），调用 Claude 根据失败情况提出改进，对新 description 同时评测训练集和测试集，最多迭代 5 次。完成后在浏览器打开 HTML 报告，返回 JSON 中的 `best_description`——按测试集评分选取，避免过拟合训练集。

**触发机制说明**：Claude 通过 `available_skills` 列表的 name + description 决定是否调用 skill，且只在自己无法轻松处理的任务上才会咨询 skill。简单的一步操作（"读这个 PDF"）即使 description 很匹配也可能不触发，因为 Claude 直接用基础工具就能搞定。复杂、多步骤、专业性强的查询，只要 description 匹配，就会可靠触发。因此评测查询应该足够实质性，让 Claude 真正需要 skill 的帮助。

### 第 4 步：应用结果

从 JSON 输出中取 `best_description`，更新 SKILL.md frontmatter。向用户展示前后对比并报告评分。
