# skill-creator-cn

**一句话：** 你说一句模糊的话，它带你把需求问清楚，写出一个能用的 skill，并告诉你怎么装、怎么用——全程中文，不用懂技术。

输入是「我想做个帮我做 X 的 skill」，产出是一个**能用的 skill 文件夹 + 你知道怎么用它**。

---

## 一、典型场景

真实世界里，"想做 skill"的人长这样：

**场景 A｜完全不懂技术，但有重复的烦事**
> 做运营的小李：「我每天都要把客户发来的报价邮件整理成一张表格，烦死了，能不能搞个东西帮我。」

他不知道 skill 是什么结构、不会写、更不懂"评测"。他只知道自己每天在重复一件烦事。

**场景 B｜刚和 Claude 做完一件事，想存下来复用**
> 「咱们刚才一起把这份问卷 CSV 分析完了，挺好用——能不能把这个过程存成一个 skill，以后我直接调？」

流程已经发生过一遍，用户不想再从头描述一遍。

**场景 C｜手上有现成的流程 / 文档**
> 「这是我们团队的客服回复 SOP（一份文档），照着它做一个 skill 吧。」

或者：「我给你两个'输入→输出'的例子，让 skill 照着这个套路来。」

**场景 D｜做过一个 skill，但不好用**
> 「我之前弄了个 skill，可它老是不被触发 / 触发了但做出来的不对，帮我看看怎么回事。」

**场景 E｜有经验的人，想快点产出高质量 skill**
> 「我要做个 skill，输入是会议录音文字稿，输出带行动项的结构化摘要，从需求到 SKILL.md 一条龙，最后帮我跑严谨的评测。」

---

## 二、痛点

这些场景背后，是几个反复出现的真实障碍：

| # | 痛点 | 谁最常遇到 |
|---|---|---|
| 1 | **想不清需求**：有个模糊念头，但说不清输出是什么、什么时候该触发、输入长什么样、哪些情况不该处理 | 几乎所有人 |
| 2 | **不懂门道、被术语吓退**：一看到 JSON、assertion、benchmark、frontmatter 就退场 | 非技术用户 |
| 3 | **不知道这事值不值得做成 skill**：可能一句话就能让 Claude 直接做，白做一个 skill | 新手 |
| 4 | **被迫从头盘问**：明明刚做过一遍 / 已有文档，却还要重新一条条交代 | 场景 B/C |
| 5 | **做完不会用**：skill 生成了，但放哪、怎么装、怎么确认会被触发——一概不知 | 新手 |
| 6 | **英文门槛**：现有工具是英文的，连跑出来的报告也是英文 | 中文用户 |
| 7 | **不知道为什么不好用**：已有 skill 出问题，但不知道是描述写得不对、还是指令含糊 | 场景 D |

现有的 skill 创作工具，大多**会写、会评测、会打包**，却恰恰在最前、最难的"帮人把需求想清楚"这一段最薄，而且是英文、默认就上重型评测——对上面这些人很不友好。

---

## 三、它怎么对症解决

每个痛点都有明确的机制兜住：

| 痛点 | 它的解法 |
|---|---|
| 1 想不清需求 | **四要素框架**：每个 skill 都收敛到「做什么 / 什么时候触发 / 输入输出 / 不该做什么」四点。**一次一问、给选项让你挑**而不是逼你凭空想；**先给 2–3 个方案**带你权衡；**YAGNI** 帮你砍到最小可用版。问完就有清晰骨架。 |
| 2 被术语吓退 | **自动识别你的水平**（从你的措辞判断，全程不打断问"你几级"），对小白用大白话、避免裸露术语；**验证默认只是"跑给你看"**，不往你脸上甩 JSON。 |
| 3 值不值得做 | **前置闸门**：动手前先判断"要不要做成 skill / 有没有现成类似的 / 是不是范围太大该拆"。该劝退就直说"这个你直接问我就行，不用做 skill"，省得你白忙。 |
| 4 被迫从头盘问 | **多入口路由**：识别你是从零想法、从对话提取、从材料、还是从草稿进来——从对话/材料进来的，它**直接挖取、回填，只问缺口**，不重新盘问。 |
| 5 做完不会用 | **"最后一公里"**：管到底——放哪（`.claude/skills/<名字>/`）、怎么确认装上了、怎么用真实话术自测会不会被触发、需要分发时怎么打包。 |
| 6 英文门槛 | **全中文**：从引导对话、生成的 skill，到评测审阅界面和优化报告，全部中文化（只有给机器读的内部契约保持英文）。 |
| 7 不知为何不好用 | **诊断剧本**：对"不触发 / 行为跑偏 / 过欠触发 / 太啰嗦"分别给「症状→检查点→动作」的排查路径，先诊断再改，而不是从头重做。 |

还有一道贯穿全程的关卡——**动手前的确认门**：四要素挖齐后，它会把理解回述给你确认，"这样对吗？没问题我就开始写了"，**理解没对齐绝不开写**，避免在误解上白跑一版。

## 四、核心价值

**最根本的一条：它把"做一个能用的 skill"这件事，从「得懂行才能干」压到了「会说中文、能回答几个问题就能干」。**

具体体现在四点：

1. **帮你想清楚，而不只是帮你写。** 它最大的价值在最前端——把模糊想法逼成清晰规格。这是别的工具最薄、而它最厚的地方。

2. **见人下菜碟，一个工具服务所有水平。** 自动适应水平 + 分层验证：小白不被术语吓退，老手不被啰嗦引导拖累——同一个工具，两头都照顾到。

3. **管到能用为止。** 不是生成完 SKILL.md 就撒手，而是一路带到"装好、跑通、确认会触发"。

4. **增强而非重造，所以既好用又可靠。** 它没有推倒成熟的官方 skill-creator 重来，而是站在它肩上——复用其经过验证的评测 / 打包 / 描述触发优化后端，只把最薄弱的前端"需求共创"重写、全程中文化、按水平分层。

> 它本身也是"好 skill"的样板：主体精简（薄路由器，细节按需加载）、用解释代替死规定、把触发描述当头等大事——它教你怎么做好 skill 的方式，就是把自己做成了那个样子。

## 五、走一遍完整流程

以场景 A（整理报价邮件）为例：

1. **你说**：「我每天把客户报价邮件整理成表格，烦死了，能不能搞个 skill。」
2. **闸门**：判断这是高频重复任务，值得做 → 继续。
3. **挖需求**（大白话，一次一问）：
   - "整理成的表格，你要的是一个能下载的文件，还是贴在对话里的内容？"（给选项）
   - "邮件里你最关心抽哪几项？金额、客户名、报价日期？"
   - "有没有哪种邮件你不想让它自动处理的？"
4. **确认门**：「我理解是——你给一封报价邮件 → skill 抽出客户名/金额/日期 → 生成一行表格数据；触发词类似'整理这封报价'。对吗？」你点头。
5. **起草**：写出 SKILL.md（含触发描述），并自检一遍。
6. **验证（轻量默认）**：拿你一封真实邮件跑一遍，把结果摆给你看 → "这样对吗？哪里不对" → 改到满意。
7. **最后一公里**：告诉你把文件夹放进 `.claude/skills/`，重开会话，用「帮我整理这封报价邮件」试触发。

全程你只是在回答问题和确认，没碰任何术语。

---

## 六、务实地说，它不做这些

- **不是一键生成**：你仍要回答几个问题、确认需求。这步省不掉——想不清的需求做不出能用的 skill。
- **一次聚焦一个 skill**：若你的想法其实是好几个 skill，它会提醒你拆开，但不自动批量生成一整套。
- **重型量化评测是进阶档**：benchmark、跨版本盲评这些它能做，但只在你是老手或主动要严谨对比时才上，不是默认体验。
- **截图/设计稿、逆向现有代码**这类参考分析，目前不支持（聚焦在"流程文档"和"输入输出例子"两类）。

---

## 谁该用它

- **完全不懂技术、却有重复痛点的人**（运营、客服、家长、个体户……）：第一次能把"我每天都要做的烦事"亲手变成一个可复用的 skill。
- **有经验、想快出高质量 skill 的人**：跳过啰嗦引导，直接给结构化需求、上严谨评测。

无论哪一种，目标都一样：**少花力气，做出一个真能用的 skill。**
